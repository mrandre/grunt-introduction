Introduction to Grunt
A Crash Course for the Curious
What is Grunt?
Grunt is a _task manager_ written in JavaScript or CoffeeScript.
Grunt makes tedious tasks fast and quick.
You can write your own tasks, but usually, you will install somebody elses and provide configuration information.
There are a LOT of grunt plugins.
Grunt is pretty easy. You just have to learn how everything works.
Grunt runs from the terminal, though you can also run it in other places.
Grunt requires a "gruntfile", a small node application that describes what plugins to use, and your various configurations. When you run grunt, it will look for the nearest gruntfile, read the provided configurations, and execute tasks.
A grunt command might look like this: "grunt build:stage"
"grunt" is the grunt application, "build" is a task, and "stage" is either a subset of configurations, or an argument to the task. We'll see how this works later, but know that "grunt task:argument" is the format for calls.
Installation
First, you will need node.js and npm; Grunt is an npm module.
The easiest way to get it is to install homebrew.
brew install node
Installing node installs npm, the node package manager.
Make a folder.
cd ~/Code/gruntdemo; cd ~/Code/gruntdemo
Initialize an npm app.
npm init
Answer the questions, mostly who you are, name of your module. You can skip anything that doesn't seem to matter.
There should now be a file called package.json. You don't need to edit it, but we'll make handy use of it.
If you have git installed, now would be a good time to boot up a git repo
git init
git add package.json
git commit -m "Added package.json file via npm init."
touch .gitignore; edit .gitignore (use your editor of choice, vim, subl, whatever)
Add line to .gitignore: node_modules
More on this in a minute.
Install grunt and grunt-cli
Grunt has two parts, the main application that runs the tasks, and a small application that runs globally and locates the nearest grunt.
Why? For one thing, over its development, Grunt has changed radically. It's hard to use apps with different grunt versions if grunt is installed globally. So instead, the app decides what version it needs and installs it locally. We get the convenience of a global install, with the reliability of a local one.
npm install grunt --save-dev
This will create a new folder called "node_modules", and place grunt and its dependencies inside that folder.
The --save-dev part is interesting. NPM will install the files, then add a section to the package.json file called "dev-dependencies", and add a line for grunt, pointing at a version number. Why is this useful? Earlier, I urged you to have git ignore the node_modules directory. These dependencies can get fairly expansive. We usually don't want them in the repository, particularly because newer versions will come out, and we usually would prefer the newer compatible version. NPM will note the version in such a way that later installs can grab minor improvements. When another developer grabs your app, if it has dependencies in the package.json, you can just run "npm install" and npm will automatically scan package.json, and load dependencies, automatically jumping to safe bug fix releases.
It's pretty cool.
sudo npm install -g grunt-cli
This installs the global grunt command. That's what -g means. Since this will install to a major directory, you'll need admin privileges to run the command. Hence the sudo.
After running this, you should be able to run "which grunt" and get a response.
grunt
If you attempt to run grunt, you should get an error, noting that you don't have a grunt file. Let's make one.
Grunt File
touch Gruntfile.js
You can also create Gruntfile.coffee and use CoffeeScript. Grunt will read coffee gruntfiles with no further effort on your part.
subl Gruntfile.js OR vim Gruntfile.js OR foo Gruntfile.js (open your new file.)
Grunt files do three main things:
Define configurations
Load Plugins
Define Tasks
More on each of these as we go.
How Grunt Works
Grunt is an app that runs other apps. Another way to think about it is as a callback. Your Gruntfile needs to feed the grunt system a callback to tell it how to deal with your app.
Since grunt is a node app, it creates callbacks in the Node module style. When you require a dependency in node, you say something like
var dependency = require(“./libs/mylib.js”);
Node looks in the local folder for libs, then mylibs.js, and executes the file. It looks for a value called “module.exports” (among others). Whatever this value is set to is returned to the variable dependency.
When grunt runs, it will execute your gruntfile, and expect to find a module.exports value that is a callback function. This callback will receive one argument, a “grunt” object with a couple of useful methods. It will look like this:
module.exports = function(grunt) {};
Or in Coffee:
module.exports = (grunt) ->
Config
The great bulk of your project will be composed of configuring tasks. Think of it like buckets; Different modules and plugins will create slots, which you can fill in with configurations.
Config is loaded into grunt with grunt.initConfig(), which takes a hash as an argument.
The first thing you’ll want to include is the contents of your package.json file. It contains name and version information that will be useful for some of your tasks. (For instance, you can use this data to create built files like my-app.0.1.8.js). To load the package, we do this:
module.exports = function(grunt) {
grunt.initConfig({
pkg: grunt.file.readJSON("package.json")
});
};
Now we’ll be able to use values like pkg.name and pkg.version, which are part of our package settings. That means we won’t duplicate the data and set ourselves up for future bugs. Handy. Let’s do something useful, now. We’re going to need a plugin.
Plugins
Plugins are bundles of grunt tasks, usually acquired using npm, or locally if of your own devisement.
To get a sense of the bredth of available plugins, type this in your terminal:
npm search grunt-
Let’s use a relatively simple one: grunt-contrib-concat
The “-contrib-“ part means the plugin is maintained by the maintainers of grunt itself. Which means it is highly reliable, and will work with new grunt versions as they come out. You can do most of your important work using only contrib plugins.
In a terminal, type:
npm install grunt-contrib-concat —save-dev
Now we need some files to concat.
mkdir src; mkdir dist; touch src/foo.js; touch src/bar.js; touch src/baz.js
Now create a function in each file. Doesn’t matter what.
We want to grab every file in our src directory, combine them into a single file, and place that file in the dist folder.
First, we need to add a “concat” section to our grunt config. Like so:
module.exports = function(grunt) {
grunt.initConfig({
pkg: grunt.file.readJSON("package.json"),

concat: {

}

});
};
The main things we need to determine is what files are read in, and what file is written out. It’s important to note that we may have multiple concat operations (grunt easily supports this) so we’ll group our settings together as “main”.
module.exports = function(grunt) {
grunt.initConfig({
pkg: grunt.file.readJSON("package.json"),

concat: {
main: {
src: [“src/**/*.js”],
dest: “dist/<%= pkg.name %>-<%= pkg.version %>.js”
}
}

});
};
A couple of things to observe:
“src” is an _array_. You can list multiple sources to pull into the same build file.
“src” uses a “glob”, a way of referring to multiple files succinctly. The “**” means “any number of subdirectories”, and the “*” means “any filename.” So we’ll take any file with a .js extension in the src directory, or any subdirectory of the src directory.
“dest” is a single file. Note the “<%= pkg.name %>”. This refers to the value in our pacakge.json file. This is like a jsp or php template insertion. Same with <%= pkg.version %>.
You can use these template insertions.
If we run
grunt concat
We get an error: Warning: Task “concat” not found.
The reason is we need to tell grunt about our plugin.
module.exports = function(grunt) {
grunt.initConfig({
pkg: grunt.file.readJSON("package.json"),

concat: {
main: {
src: ["src/**/*.js"],
dest: "dist/<%= pkg.name %>-<%= pkg.version %>.js"
}
}

});

grunt.loadNpmTasks("grunt-contrib-concat");
};
Now that we’ve added a concatination plugin, let’s add minification with the uglify plugin. First, we’ll install the plugin.
npm install grunt-contrib-uglify —save-dev
Next, we’ll tell grunt about our plugin.
module.exports = function(grunt) {
grunt.initConfig({
pkg: grunt.file.readJSON("package.json"),

concat: {
main: {
src: ["src/**/*.js"],
dest: "dist/<%= pkg.name %>-<%= pkg.version %>.js"
}
}

});

grunt.loadNpmTasks("grunt-contrib-concat");
grunt.loadNpmTasks("grunt-contrib-uglify");
};
And finally, our config.
module.exports = function(grunt) {
grunt.initConfig({
pkg: grunt.file.readJSON("package.json"),

concat: {
main: {
src: ["src/**/*.js"],
dest: "dist/<%= pkg.name %>-<%= pkg.version %>.js"
}
},

uglify: {
main: {
src: ["<%= concat.main.dest %>"],
dest: "dist/<%= pkg.name %>-<%= pkg.version %>.min.js"
}
}

});

grunt.loadNpmTasks("grunt-contrib-concat");
grunt.loadNpmTasks("grunt-contrib-uglify");
};
Of note:
Our source value is “<%= concat.main.dest =%>”.
This means “Use the config value of concat task, main section, value ‘dest’.” Think about how handy this is. Our changes will cascade downward if we change them. It’s important that we always pick up the right file, and this way, we can be sure we always will. Handy!
Let’s make one small improvement. Notice those two calls to “loadNpmTasks” at the bottom? It may not seem like it now, but once we start using a lot of plugins, it’s going to get tedious, adding that same line over and over. It would be nice to automate this. We can do it with a plugin called “load-grunt-tasks”.
Go to your terminal and type
npm install load-grunt-tasks —save-dev
Now we’re going to make a few changes to our gruntfile
module.exports = function(grunt) {
require("load-grunt-tasks")(grunt);

grunt.initConfig({
pkg: grunt.file.readJSON("package.json"),

concat: {
main: {
src: ["src/**/*.js"],
dest: "dist/<%= pkg.name %>-<%= pkg.version %>.js"
}
},

uglify: {
main: {
src: ["<%= concat.main.dest %>"],
dest: "dist/<%= pkg.name %>-<%= pkg.version %>.min.js"
}
}

});
};
The big change is at the top:
require("load-grunt-tasks")(grunt);
We also removed all the calls to grunt.loadNpmTasks from the bottom. “load-grunt-tags” handles this for us, scanning node_modules for plugins of the sort “grunt-*” and loading them for you. Super Handy.
Tasks
Most of the time, we will want to run both of our tasks, “concat” and “uglify” together. The way to do this from the command line is
grunt concat uglify
That’s a lot of typing. It would be better to just type “build” and have that call both. The way to do this is with the registerTask method. We can add one line to our file:
module.exports = function(grunt) {
require("load-grunt-tasks")(grunt);

grunt.initConfig({
pkg: grunt.file.readJSON("package.json"),

concat: {
main: {
src: ["src/**/*.js"],
dest: "dist/<%= pkg.name %>-<%= pkg.version %>.js"
}
},

uglify: {
main: {
src: ["<%= concat.main.dest %>"],
dest: "dist/<%= pkg.name %>-<%= pkg.version %>.min.js"
}
}

});

grunt.registerTask("build", ["concat", "uglify"]);
};
Notice that line at the bottom. It’s just an alias. “when I say ‘build’, you run ‘concat’, then ‘uglify’”. So we can run both commands by entering:
grunt build
And now we are running both. Of course, for this grunt file, this is the main job we want to do. So we could be even lazier, and make this task the default task.
module.exports = function(grunt) {
require("load-grunt-tasks")(grunt);

grunt.initConfig({
pkg: grunt.file.readJSON("package.json"),

concat: {
main: {
src: ["src/**/*.js"],
dest: "dist/<%= pkg.name %>-<%= pkg.version %>.js"
}
},

uglify: {
main: {
src: ["<%= concat.main.dest %>"],
dest: "dist/<%= pkg.name %>-<%= pkg.version %>.min.js"
}
}

});

grunt.registerTask("build", ["concat", "uglify"]);
grunt.registerTask("default", "build");
};
We could have just defined concat and uglify as default, but I like the slight layer of separation. Over time, you’ll probably add many plugins and define many tasks. You may find it useful to group them into sub-tasks so you can run them separately. Ultimately, it’s up to you. Now that we’ve defined a default task, we can build and minify using
grunt
Coffee
I mentioned I usually write my grunt files in coffeescript. Here’s what ours would look like in Coffee:
module.exports = (grunt) ->
require("load-grunt-tasks")(grunt)

grunt.initConfig(
pkg: grunt.file.readJSON("package.json")

concat:
main:
src: ["src/**/*.js"]
dest: "dist/<%= pkg.name %>-<%= pkg.version %>.js"

uglify:
main:
src: ["<%= concat.main.dest %>"]
dest: "dist/<%= pkg.name %>-<%= pkg.version %>.min.js"

)

grunt.registerTask("build", ["concat", "uglify"])
grunt.registerTask("default", "build")
For a small file like this, the differences are less noticable, but still recognizable. For a config file like this, Coffee is a lot cleaner, and produces fewer lines of code. Ordinarily, CoffeeScript has the tradeoff of requiring process to use, but in this case, Grunt can handle coffee natively.
I recommend you consider it for your own uses. You can learn more about CoffeeScript at http://coffeescript.org/
